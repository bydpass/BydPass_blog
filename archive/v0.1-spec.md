# 需求描述

一句话 goal：

> **建立一个轻量、可维护、适合未来接入 LLM librarian 的 GitHub 信息收集 repo，用来集中归档、整理、复用来自小红书/网页/视频/其他来源的收藏与种草信息。**

下面这份可以直接丢给 Claude Code。🧩

````md
# GitHub 信息收集仓库需求说明 v0.1

## 1. 项目目标

建立一个轻量级 GitHub repo，用于集中收集、归档、整理和复用各类信息。

当前主要用途：

- 保存小红书收藏与种草内容
- 保存网页、视频、文章、工具、教程等信息来源
- 将零散收藏转化为可检索、可维护、可复用的 Markdown 知识库
- 为未来接入 LLM librarian / 自动整理脚本 / Obsidian 联动预留结构

本项目优先追求：

1. 简单
2. 可读
3. 可扩展
4. 低维护成本
5. 对 GitHub 新手友好

暂不追求复杂自动化。

---

## 2. 核心设计原则

### 2.1 Inbox First

所有新内容先进入 `inbox/`，不要一开始强行分类。

理由：

- 降低收集阻力
- 避免分类焦虑
- 方便后续批量整理
- 适合未来交给 LLM librarian 处理

---

### 2.2 Markdown First

所有核心笔记使用 Markdown。

理由：

- GitHub 原生支持
- 可读性强
- 易于版本管理
- 可被 Obsidian、VS Code、Claude Code、ChatGPT 等工具读取

---

### 2.3 Assets Separate

图片、截图、PDF、附件等原始素材统一放进 `assets/`。

理由：

- 避免 Markdown 文件夹膨胀
- 方便后续迁移
- 方便引用和维护

---

### 2.4 Manual Before Automation

v0.1 阶段先跑通手动流程，不急着写爬虫、脚本或复杂 bot。

初始验收标准：

> 能够手动收集、记录、整理至少 20 条信息，并保持结构清晰。

---

## 3. 建议目录结构

```text
repo-name/
├── README.md
├── inbox/
│   └── 2026-05.md
├── notes/
│   ├── products/
│   ├── tools/
│   ├── ideas/
│   ├── tutorials/
│   └── references/
├── assets/
│   ├── images/
│   └── files/
├── tags.md
└── archive/
````

---

## 4. 各目录职责

### 4.1 `README.md`

仓库说明文件。

需要包含：

* 仓库用途
* 使用方式
* 收录范围
* 基本目录说明
* 当前工作流
* 后续计划

---

### 4.2 `inbox/`

临时收集箱。

用于存放尚未整理的原始信息。

建议按月份建立文件：

```text
inbox/
├── 2026-05.md
├── 2026-06.md
└── 2026-07.md
```

每条信息先进入当月 inbox。

---

### 4.3 `notes/`

整理后的正式笔记区。

建议分类：

```text
notes/
├── products/      # 商品、种草、消费决策
├── tools/         # 软件、网站、工具、工作流
├── ideas/         # 灵感、观点、可发展想法
├── tutorials/     # 教程、操作指南、学习资料
└── references/    # 参考资料、文章、论文、背景材料
```

---

### 4.4 `assets/`

素材区。

建议结构：

```text
assets/
├── images/        # 图片、截图
└── files/         # PDF、文档、附件
```

命名建议：

```text
YYYY-MM-DD-short-description.ext
```

例：

```text
2026-05-17-xhs-keyboard-layout.png
2026-05-17-github-guide.pdf
```

---

### 4.5 `tags.md`

标签系统说明。

用于统一标签命名，避免标签野蛮生长。

建议初始标签：

```md
# Tags

## 来源
- #source/xiaohongshu
- #source/web
- #source/video
- #source/github
- #source/book

## 类型
- #type/product
- #type/tool
- #type/tutorial
- #type/idea
- #type/reference

## 状态
- #status/raw
- #status/processed
- #status/archived
- #status/action-needed

## 价值
- #value/buy-later
- #value/learn-later
- #value/use-now
- #value/research
```

---

### 4.6 `archive/`

归档区。

用于存放：

* 过时内容
* 已废弃方案
* 不再活跃但不想删除的信息
* 被整理后不再需要留在 inbox 的旧内容

---

## 5. Inbox 条目模板

在 `inbox/YYYY-MM.md` 中使用以下格式：

```md
## 标题

- 来源：
- 链接：
- 类型：
- 标签：
- 收集日期：
- 一句话价值：
- 原始摘录：
- 我的判断：
- 后续动作：
```

示例：

```md
## Mac 清理脚本优化思路

- 来源：ChatGPT / 自己实践
- 链接：
- 类型：工具 / 工作流
- 标签：#type/tool #status/raw #value/use-now
- 收集日期：2026-05-17
- 一句话价值：可以作为未来维护 macOS 环境卫生的基础脚本。
- 原始摘录：
  - dry-run
  - whitelist
  - cache cleanup
- 我的判断：
  这个方向值得保留，但脚本必须保持 caveman-readable，不能变成看不懂的邪术。
- 后续动作：
  整理进 notes/tools/mac-cleaning-script.md
```

## 7. README.md 初始内容要求

`README.md` 应该包含以下结构：

````md
# 信息收集仓库

这是一个用于集中收集、归档、整理和复用信息的个人知识仓库。

## 目标

将零散收藏转化为可检索、可维护、可复用的 Markdown 信息系统。

## 当前用途

- 小红书收藏整理
- 商品与工具种草记录
- 教程与文章归档
- 灵感与想法沉淀
- 为未来 LLM librarian 预留结构

## 工作流

1. 新信息先进入 `inbox/`
2. 每隔一段时间批量整理
3. 有价值内容移动到 `notes/`
4. 图片、PDF、附件放入 `assets/`
5. 过时内容放入 `archive/`

## 目录结构

见下方：

```text
repo-name/
├── README.md
├── inbox/
├── notes/
├── assets/
├── tags.md
└── archive/
````

## 10. 非目标

v0.1 阶段不做：

* 自动爬取小红书
* 自动下载图片
* 自动摘要
* 自动分类
* GitHub Actions
* 数据库
* Web App
* 浏览器插件
* 复杂脚本
* 多人协作权限设计

## 12.执行口径

请按以下原则执行：

1. 只创建必要目录和 Markdown 文件。
2. 文件内容要让 GitHub 新手看得懂。
3. 注释和说明使用中文。
4. 不要过度工程化。
5. 不要主动添加依赖、脚本或自动化。
6. 保持结构稳定，方便未来扩展。
